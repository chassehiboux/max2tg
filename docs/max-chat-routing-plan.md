# План: отдельная Telegram-группа для каждого чата MAX

## Цель

Перевести проект с модели `все MAX-чаты → один TG_CHAT_ID` на модель `один MAX-чат → одна Telegram-группа`, сохранив текущую возможность отвечать обратно в MAX.

## Этапы

### Этап 1. Подготовить новую модель конфигурации и состояния

Что изменится для пользователя:

- бот будет знать, кто является владельцем;
- список привязок будет храниться локально и переживать перезапуск;
- ручная правка `.env` для каждой новой группы больше не понадобится.

Что сделать технически:

- заменить обязательный `TG_CHAT_ID` на `TG_ADMIN_ID`;
- добавить слой локального хранилища JSON;
- описать статусы чата: `unconfigured`, `bound`, `muted`, `pending_bot`.

### Этап 2. Вынести технические сообщения в личку владельца

Что изменится для пользователя:

- статусы подключения и ошибки перестанут засорять рабочие группы;
- личка с ботом станет точкой управления.

Что сделать технически:

- разделить отправку технических сообщений и отправку рабочих сообщений;
- направлять connect/disconnect/error только в `TG_ADMIN_ID`;
- запускать Telegram Application всегда.

### Этап 3. Сделать интерфейс управления чатами

Что изменится для пользователя:

- появится кнопка `Настроить чаты`;
- можно будет привязывать, отвязывать и отключать чаты прямо из Telegram;
- новые MAX-чаты будут требовать решения через кнопки.

Что сделать технически:

- добавить команды и callback-обработчики для личного чата;
- добавить выбор Telegram-группы через `request_chat`;
- сохранить привязку выбранного `chat_id` в JSON;
- обновлять статус чата после выбора группы.

### Этап 4. Переделать маршрутизацию доставки

Что изменится для пользователя:

- каждый MAX-чат будет приходить только в свою Telegram-группу;
- неотслеживаемые чаты перестанут попадать в Telegram;
- у привязанных, но ещё недоступных групп будет понятный статус ожидания, а сообщения не потеряются.

Что сделать технически:

- перестроить `TelegramSender` на отправку в динамический `chat_id`;
- на каждом входящем сообщении выбирать целевую группу по локальной привязке;
- для `pending_bot` держать локальную очередь и дозаливать её после появления доступа к группе;
- аккуратно обработать ошибки `Forbidden`/`Chat not found`/`Bot is not a member`.

### Этап 5. Ограничить ответы обратно в MAX владельцем

Что изменится для пользователя:

- reply-кнопки в группах останутся;
- отвечать в MAX сможет только владелец;
- другие участники группы не смогут случайно отправить что-то в MAX.

Что сделать технически:

- во всех callback и text handlers проверять `effective_user.id == TG_ADMIN_ID`;
- состояния ответа хранить так, чтобы сценарии из разных групп не конфликтовали;
- оставить текущий `REPLY` payload в MAX без изменения бизнес-логики.

### Этап 6. Научить бота обнаруживать новые чаты MAX

Что изменится для пользователя:

- при появлении нового чата не нужно следить за логами или редактировать конфиг;
- бот сам напомнит, что появился новый источник сообщений.

Что сделать технически:

- сравнивать snapshot MAX и локальное хранилище;
- при первом сообщении из неизвестного чата создавать локальную запись;
- отправлять одноразовое уведомление в личку с кнопками `Привязать` и `Не отслеживать`.

### Этап 7. Проверки и документация

Что изменится для пользователя:

- поведение будет воспроизводимым после обновления;
- запуск и первичная настройка останутся понятными.

Что сделать технически:

- обновить тесты `config`, `tg_handler`, `max_listener`, при необходимости добавить тесты для нового хранилища;
- обновить `.env.example` и README;
- описать ручной сценарий первичной привязки и повторного запуска.

## Риски

### Выбор группы без добавления бота

Риск:

- пользователь выбрал группу, но бот не может туда писать.

Снижение риска:

- хранить привязку отдельно от факта доставки;
- явно показывать статус `Ожидает добавления бота`;
- уведомлять владельца при первой неудачной попытке доставки.

### Перепутанные состояния ответа

Риск:

- если владелец одновременно отвечает в нескольких группах, состояния могут смешаться.

Снижение риска:

- хранить состояние reply-сценария с учётом Telegram-чата и пользователя;
- очищать завершённые сценарии сразу после отправки или отмены.

### Рост очереди недоставленных сообщений

Риск:

- если бот долго не добавляют в группу, локальная очередь может расти.

Снижение риска:

- ввести понятный лимит на очередь;
- уведомлять владельца о накоплении;
- документировать это поведение.

## Зависимости

- Telegram Bot API chat selection через `request_chat`;
- существующий MAX WebSocket-клиент;
- локальная файловая система для JSON-хранилища.
