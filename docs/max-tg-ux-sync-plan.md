# План работ: улучшение UX ответов MAX ↔ Telegram

## Пользовательский результат

После завершения согласованного объёма переписка будет выглядеть так:

- в MAX уйдёт чистый текст без телеграмной служебной шапки;
- временные подсказки в Telegram не будут зависать;
- успешный ответ будет виден прямо внутри исходного сообщения, а не отдельным служебным уведомлением;
- при ответе можно будет выбрать, отправлять ли обычное сообщение или reply к конкретному сообщению.

## Этапы

### Этап 1. Очистить исходящий текст в MAX

Что меняем:

- убираем автопрефикс `💬 Имя:` из ответа Telegram → MAX;
- сохраняем текущую логику отправки без дополнительных UI-изменений.

Почему первым:

- минимальный риск;
- быстрый видимый результат;
- не зависит от хранения дополнительного состояния.

### Этап 2. Сделать единый чистый UX ответа в Telegram

Что меняем:

- после нажатия кнопки ответа сохраняем ID подсказки;
- после отправки ответа или отмены удаляем подсказку;
- сохраняем ID исходного телеграм-сообщения, под которым нажата кнопка;
- после успешной отправки редактируем именно его;
- добавляем в конец блок ответа.

Что это даёт:

- чат становится чище;
- сценарий ответа выглядит как одно короткое действие, а не как ручная работа с ботом;
- история остаётся компактной и читаемой.

Что важно предусмотреть:

- повторные ответы к одному сообщению;
- лимиты Telegram на редактирование текста и caption;
- fallback для не редактируемых типов сообщений.

### Этап 3. Добавить выбор режима ответа

Что меняем:

- кнопки под сообщением не используются;
- пользователь выбирает поведение через обычное сообщение в топике или стандартный Telegram reply.

Предлагаемый UX:

- написать текст в нужный топик;
- сделать Telegram reply на пересланное сообщение MAX.

Почему последним:

- этот этап использует данные о конкретном `message_id` исходного сообщения;
- нужно отдельно проверить формат отправки reply в MAX на живом веб-клиенте.

Как проверяем:

- используем `playwright` с сохранённым браузерным контекстом;
- открываем `web.max.ru`;
- вы входите в аккаунт вручную;
- после авторизации я смотрю реальное поведение reply в чате, например с `Анна Безверхая`.

## Идеи и рекомендации

### Идея 1. Сразу заложить единый объект состояния ответа

Вместо хранения только `chat_id` лучше хранить одну структуру ожидания ответа:

- `max_chat_id`
- `max_message_id` для reply-режима
- `mode` (`message` или `reply`)
- `prompt_message_id`
- `source_telegram_message_id`
- `source_label`

Это упростит блок задач 2 и 3, а также задачу 5.

### Идея 2. Для редактирования Telegram-сообщений расширить `TelegramSender`

Сейчас отправка в Telegram не возвращает объект сообщения наружу. Для блока задач 2 и 3 лучше:

- возвращать `telegram.Message` из методов отправки;
- добавить методы редактирования текста/caption;
- централизовать fallback там же.

### Идея 3. Для задачи 5 сначала поймать реальный payload reply из MAX

Перед полной реализацией полезно один раз зафиксировать реальный пример отправки reply, чтобы не угадывать структуру пакета. Это снизит риск сделать поддержку “почти правильно”, но на неверном поле.

## Решения по умолчанию, если ничего не уточнять

1. Задачу 1 делаем безусловно: полностью убираем префикс имени.
2. В блоке задач 2 и 3 подсказку удаляем и после успешной отправки, и после `/cancel`.
3. В блоке задач 2 и 3 основной сценарий — текстовые сообщения и медиа с caption.
4. Задачу 5 начинаем с UX выбора; поддержку именно MAX reply включаем после подтверждения формата payload через `playwright`.
