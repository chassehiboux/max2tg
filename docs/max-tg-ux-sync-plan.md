# План работ: улучшение UX ответов и синхронизации MAX ↔ Telegram

## Пользовательский результат

После завершения всех пяти задач переписка будет выглядеть так:

- в MAX уйдёт чистый текст без телеграмной служебной шапки;
- временные подсказки в Telegram не будут зависать;
- успешный ответ будет виден прямо внутри исходного сообщения, а не отдельным служебным уведомлением;
- правки сообщений в MAX будут доходить до Telegram;
- при ответе можно будет выбрать, отправлять ли обычное сообщение или reply к конкретному сообщению.

## Этапы

### Этап 1. Очистить исходящий текст в MAX

Что меняем:

- убираем автопрефикс `💬 Имя:` из ответа Telegram → MAX;
- сохраняем текущую логику отправки без дополнительных UI-изменений.

Почему первым:

- минимальный риск;
- быстрый видимый результат;
- не зависит от хранения ID сообщений.

### Этап 2. Убрать временный технический мусор в Telegram

Что меняем:

- после нажатия кнопки ответа сохраняем ID подсказки;
- после отправки ответа или отмены удаляем подсказку.

Что это даёт:

- чат становится чище;
- сценарий ответа выглядит как одно короткое действие, а не как ручная работа с ботом.

### Этап 3. Заменить `✅ Отправлено` на редактирование исходного сообщения

Что меняем:

- сохраняем ID исходного телеграм-сообщения, под которым нажата кнопка;
- после успешной отправки редактируем именно его;
- добавляем в конец блок ответа.

Что важно предусмотреть:

- повторные ответы к одному сообщению;
- лимиты Telegram на редактирование текста и caption;
- fallback для не редактируемых типов сообщений.

### Этап 4. Добавить реестр соответствий сообщений и синхронизацию правок из MAX

Что меняем:

- при пересылке входящего сообщения из MAX сохраняем связь между ID сообщения в MAX и ID сообщения в Telegram;
- при событии редактирования из MAX обновляем соответствующее сообщение в Telegram;
- показываем пометку `(ред.)`.

Базовое допущение:

- на первом этапе реестр живёт в памяти;
- после рестарта старые маппинги не восстанавливаются.

### Этап 5. Добавить выбор режима ответа

Что меняем:

- кнопки под сообщением становятся более явными;
- пользователь выбирает: обычное сообщение или reply к конкретному сообщению MAX.

Предлагаемый UX:

- `📨 Сообщение`
- `↩️ Reply`

Почему последним:

- этот этап использует данные о конкретном `message_id` исходного сообщения;
- нужно отдельно проверить формат отправки reply в MAX.

## Идеи и рекомендации

### Идея 1. Сразу заложить единый объект состояния ответа

Вместо хранения только `chat_id` лучше хранить одну структуру ожидания ответа:

- `max_chat_id`
- `max_message_id` для reply-режима
- `mode` (`message` или `reply`)
- `prompt_message_id`
- `source_telegram_message_id`
- `source_label`

Это упростит задачи 2, 3 и 5.

### Идея 2. Для редактирования Telegram-сообщений расширить `TelegramSender`

Сейчас отправка в Telegram не возвращает объект сообщения наружу. Для задач 3 и 4 лучше:

- возвращать `telegram.Message` из методов отправки;
- добавить методы редактирования текста/caption;
- централизовать fallback там же.

### Идея 3. Для задачи 4 сначала поймать реальный payload редактирования из MAX

Перед полной реализацией полезно один раз зафиксировать реальный пример события редактирования, чтобы не угадывать структуру пакета. Это снизит риск сделать поддержку “почти правильно”, но на неверном поле.

### Идея 4. Не дублировать пометку `(ред.)`

Если сообщение редактируют несколько раз, пометку лучше хранить как часть форматирования, а не просто бесконечно дописывать строку `(ред.)`.

## Решения по умолчанию, если ничего не уточнять

1. Задачу 1 делаем безусловно: полностью убираем префикс имени.
2. Подсказку из задачи 2 удаляем и после успешной отправки, и после `/cancel`.
3. В задаче 3 основной сценарий — текстовые сообщения и медиа с caption.
4. Реестр соответствий сообщений в первой версии храним только в памяти.
5. Задачу 5 начинаем с UX выбора; поддержку именно MAX reply включаем после подтверждения формата payload.
