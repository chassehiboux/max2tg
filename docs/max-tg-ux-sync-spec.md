# Спецификация: улучшение UX ответов MAX ↔ Telegram

## Зачем это нужно

Сейчас бот уже умеет:

- пересылать входящие сообщения из MAX в Telegram;
- отправлять текст из Telegram обратно в нужный чат MAX из соответствующего топика;
- отправлять reply в MAX через стандартный Telegram reply на пересланное сообщение.

Но текущий сценарий выглядит техническим, а не переписочным:

- в MAX к ответу добавляется служебная приписка с именем пользователя Telegram;
- в Telegram после нажатия кнопки остаётся временная подсказка;
- после отправки появляется отдельное сообщение `✅ Отправлено`, которое засоряет диалог;
- нет выбора между обычным сообщением в чат MAX и ответом именно на конкретное сообщение.

Цель доработки: сделать поведение ближе к обычной переписке, чтобы Telegram-чат выглядел как живая история сообщений, а не как набор технических уведомлений.

## Текущее поведение

1. Пользователь пишет текст в нужном Telegram-топике.
2. Для ответа на конкретное сообщение пользователь делает обычный Telegram reply.
3. В группах Telegram бот отправляет в MAX не только текст, но и префикс вида:
   `💬 Danil Zuikevich:`
4. В Telegram приходит отдельное подтверждение:
   `✅ Отправлено → ...`

## Зафиксированный объём работ

### Задача 1. Убрать имя пользователя Telegram из текста, который уходит в MAX

#### Как понял задачу

Когда пользователь отвечает из Telegram, в MAX должен уходить только сам текст ответа, без дополнительной строки с именем автора из Telegram.

Было:

```text
💬 Danil Zuikevich:
Привет
```

Должно стать:

```text
Привет
```

#### Что увидит пользователь

Получатель в MAX увидит чистый текст, без телеграмной служебной шапки.

#### Приёмка

- при ответе из личного чата Telegram в MAX уходит только текст;
- при ответе из группы Telegram в MAX тоже уходит только текст;
- старый формат с `💬 Имя:` больше не используется.

#### Замечание

Побочный эффект: если одним ботом пользуются несколько людей в группе Telegram, по самому тексту в MAX уже не будет видно, кто именно отправил ответ. По умолчанию фиксирую поведение именно так, потому что это прямое требование задачи.

### Блок задач 2 и 3. Очистить UX ответа в Telegram

Этот блок объединяет две связанные доработки:

- временная подсказка должна исчезать после завершения сценария;
- вместо отдельного `✅ Отправлено` ответ должен встраиваться прямо в исходное сообщение.

#### Часть 2. Удалять временную подсказку после ответа пользователя

#### Как понял задачу

После того как пользователь нажал кнопку ответа и получил подсказку:

```text
✏️ Напишите ответ для ✉ Анна Безверхая (ответом на оригинальное сообщение):
(или /cancel для отмены)
```

эта подсказка не должна оставаться висеть в чате после того, как пользователь уже отправил свой ответ.

#### Что увидит пользователь

- бот покажет подсказку только на время ввода ответа;
- после отправки ответа подсказка исчезнет;
- в чате не останется лишнего технического мусора.

#### Приёмка

- бот запоминает ID служебной подсказки;
- после первого ответа пользователя подсказка удаляется;
- если пользователь нажал `/cancel`, подсказка тоже удаляется;
- если подсказка уже удалена вручную, бот не падает.

#### Идея

Лучше удалять именно сообщение-подсказку, а не сообщение пользователя. Сообщение пользователя остаётся как нормальная часть истории.

#### Часть 3. Вместо `✅ Отправлено` редактировать исходное сообщение из MAX и добавлять под ним блок с ответом

#### Как понял задачу

После отправки ответа не нужно слать отдельное техническое подтверждение. Вместо этого бот должен изменить исходное сообщение, на которое отвечали, и дописать под ним новый блок с текстом ответа.

Пример:

Было:

```text
✉️ Анна Безверхая
Привет
```

Стало:

```text
✉️ Анна Безверхая
Привет

📩 Danil Zuikevich
Привет Анна
```

#### Что увидит пользователь

- история будет компактной;
- ответ будет визуально привязан к сообщению MAX, на которое он был дан;
- отдельные подтверждения `✅ Отправлено` исчезнут.

#### Приёмка

- при успешной отправке в MAX исходное телеграм-сообщение редактируется;
- внизу добавляется блок ответа;
- техническое сообщение `✅ Отправлено → ...` больше не отправляется;
- если на одно сообщение ответили несколько раз, блоки добавляются по порядку;
- если Telegram не даёт отредактировать конкретный тип сообщения, нужен безопасный fallback.

#### Идея

Для первого этапа считать основным сценарием текстовые сообщения и медиа с подписью. Для типов, которые Telegram не даёт нормально редактировать, можно временно оставить fallback, чтобы не терять подтверждение совсем.

### Задача 5. Оставить выбор режима без кнопок под сообщением

#### Как понял задачу

При ответе на сообщение из MAX пользователь выбирает режим стандартным поведением Telegram:

- отправить обычное новое сообщение в тот же чат MAX;
- отправить именно reply на конкретное сообщение MAX.

#### Что увидит пользователь

Под входящими сообщениями из MAX нет дополнительных кнопок. Обычное сообщение в топике уходит как новое сообщение в MAX, а Telegram reply уходит как reply в MAX.

#### Предлагаемый пользовательский сценарий

Пользователь пишет в нужный топик или отвечает через стандартный Telegram reply на пересланное сообщение из MAX.

#### Приёмка

- бот различает два режима отправки;
- отдельное состояние ожидания ответа после нажатия кнопки не используется;
- для Telegram reply используется сохранённый ID исходного сообщения MAX;
- после отправки в MAX используется правильный тип связи с исходным сообщением.

#### Ограничение

Для полного режима reply в MAX нужно отдельно подтвердить точный формат исходящего payload, который привязывает новое сообщение к конкретному `message_id`.

#### Как будем проверять задачу 5

Для этой задачи не будем угадывать протокол reply только по исходникам. Зафиксирован практический способ проверки:

1. Открыть `web.max.ru` через `playwright` в сохранённом браузерном контексте.
2. Вы авторизуетесь вручную в этом окне.
3. После входа я открою нужный чат и посмотрю, как веб-клиент MAX технически отправляет reply.
4. В качестве рабочего примера можно использовать чат с `Анна Безверхая`.

Цель этой проверки: увидеть реальные сетевые данные и после этого реализовать режим reply без догадок.

## Что исключено из текущего объёма

Задача про синхронизацию редактирования уже отправленных сообщений из MAX в Telegram исключена из текущей итерации.

## Технические допущения на согласование

1. Для задачи 1 дополнительное состояние не нужно.
2. Для блока задач 2 и 3 достаточно хранить состояние текущего сценария ответа в Telegram.
3. Для задачи 5 понадобится сохранять `mode` и `max_message_id` исходного сообщения в состоянии ожидания ответа.
4. Постоянное хранилище соответствий сообщений для текущего объёма не требуется.

## Что предлагаю как рабочий порядок

1. Сначала сделать задачу 1, потому что она изолированная и не требует перестройки архитектуры.
2. Затем задачи 2 и 3 одним пакетом, потому что обе меняют UX ответа в Telegram.
3. Последней делать задачу 5, потому что она зависит от формата reply в MAX и будет подтверждаться через живой сценарий в `web.max.ru`.
