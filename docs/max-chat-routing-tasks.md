# Задачи: маршрутизация чатов MAX по Telegram-группам

## Статус

- Согласование требований: завершено
- Реализация: завершена
- Текущий фокус: нет

## Чек-лист

### 1. Перевести конфиг на модель владельца

Статус: `done`

- заменить обязательный `TG_CHAT_ID` на `TG_ADMIN_ID`;
- обновить загрузку настроек и валидацию;
- обновить `.env.example`;
- обновить `launcher.py`, если он подсказывает обязательные переменные;
- оставить понятную ошибку запуска, если задан старый конфиг без `TG_ADMIN_ID`.

### 2. Добавить локальное JSON-хранилище привязок

Статус: `done`

- создать модель записи для чата MAX;
- хранить статусы `unconfigured`, `bound`, `muted`, `pending_bot`;
- сохранять и читать JSON безопасно в UTF-8;
- покрыть чтение/запись и миграцию тестами.

### 3. Разделить технические сообщения и рабочую пересылку

Статус: `done`

- все connect/disconnect/error/status сообщения слать только владельцу;
- рабочие сообщения MAX отправлять в целевую группу по привязке;
- убрать предположение, что у отправителя всегда один фиксированный `chat_id`.

### 4. Сделать Telegram-интерфейс управления

Статус: `done`

- добавить команду входа в управление;
- показывать кнопку `Настроить чаты`;
- рендерить список чатов MAX с текущими статусами;
- добавить callback для `Привязать`, `Не отслеживать`, `Включить обратно`, `Отвязать`, `Сменить группу`;
- обработать `request_chat` и сохранение выбранной группы.

### 5. Ограничить управление и ответы владельцем

Статус: `done`

- принимать команды только от `TG_ADMIN_ID`;
- принимать reply/callback в группах только от владельца;
- давать безопасный отказ другим участникам.

### 6. Обнаруживать новые чаты MAX

Статус: `done`

- сравнивать snapshot и локальное хранилище;
- создавать запись для нового чата при первом обнаружении;
- уведомлять владельца в личке;
- не спамить повторными уведомлениями для одного и того же нового чата.

### 7. Обработать сценарий `группа выбрана, бот ещё не добавлен`

Статус: `done`

- складывать сообщения, пришедшие в период ожидания, в локальную очередь;
- хранить признак недоступности группы;
- повторно проверять доступ после выбора группы и при новых сообщениях;
- дозаливать очередь после появления доступа;
- уведомлять владельца, когда пересылка реально стала доступна.

### 8. Обновить reply-поток MAX

Статус: `done`

- сохранить текущие inline-кнопки под рабочими сообщениями;
- привязать reply-состояние к конкретной Telegram-группе;
- не ломать текущую отправку `REPLY` в MAX;
- расширить тесты на сценарии ответа из группы.

### 9. Документация и финальные проверки

Статус: `done`

- обновить README под новую модель настройки;
- описать ручной сценарий первой привязки;
- прогнать точечные тесты по изменённым модулям;
- перед коммитом проверить состав staged-файлов и не включить чужие логи.
